## Context

NotebookLM-Lite is currently a FastAPI + React/Vite NotebookLM-style workbench with selected sources, RAG chat, Notes, Studio artifacts, podcast generation, LiteLLM-backed text configuration, SeekDB-backed persistence, and a PPT placeholder. The intended product direction is to support most NotebookLM Studio capabilities; slide decks should become a real capability rather than a placeholder.

The integration source is `/Users/lzj/proj/notebook/new_pro/AIPPT`. That repo already contains a working slide-deck pipeline with:

- `src/models.py`: `DeckOutline`, `SlideOutline`, `SlidePromptPlanSet`, `SlidePromptPlan`, `PromptData`, `SlidePrompt`, `GenerationResult`.
- `src/prompt_generator.py`: source-to-outline, outline-to-prompt-plan, retry, validation, and JSON/Pydantic parsing logic.
- `src/model_router.py`: prompt/image/edit model routing.
- `api/routes/generate.py`: outline, prompt plan, SSE image generation, confirmed prompt handling.
- `api/routes/edit.py`: single-image edit route.
- `api/routes/export.py`: PDF/PPTX export, with PPTX generated by placing each slide image full-page.
- `web/src/components/DesignWorkflowPanel.tsx`, `SlideList.tsx`, `EditPanel.tsx`, `ExportButton.tsx`: useful interaction structure.
- `web/src/contexts/AppStateContext.tsx` and `web/src/services/projectStore.ts`: useful state concepts but not acceptable as the integrated persistence boundary because NotebookLM-Lite must own backend persistence.

`/Users/lzj/proj/notebook/OpenNotebookLM-AIPPT` is a separate branch/implementation. It must not be treated as the source of truth for this change.

## Goals / Non-Goals

**Goals:**

- Make Slide Deck a first-class NotebookLM-Lite Studio artifact.
- Use selected NotebookLM-Lite sources as the authoritative input. Do not create a separate PPT upload silo.
- Generate source-grounded deck outline and slide prompt plan with structured Pydantic validation.
- Require user confirmation/editing after outline generation and again after prompt-plan generation.
- Generate real slide images in Phase 1 using configured image model credentials.
- Support single-slide regenerate and single-slide edit in Phase 1.
- Export generated decks as PPTX files and allow download from the integrated UI.
- Persist all deck state in NotebookLM-Lite backend storage, not browser IndexedDB.
- Use NotebookLM-Lite model configuration for text, image, and edit model calls.
- Validate through real model execution and browser mouse-click smoke tests.

**Non-Goals:**

- Do not promise native editable PowerPoint shapes.
- Do not preserve AIPPT's independent FastAPI app or frontend app inside NotebookLM-Lite.
- Do not add a second model configuration UI for PPT.
- Do not implement video overview, infographics, or Deep Research as part of this slide deck integration.

## Architecture

### Ownership Boundary

NotebookLM-Lite owns the integrated feature:

```text
NotebookLM-Lite selected sources
        |
        v
SlideDeckService / SlideDeckWorkflow
        |
        +-- LiteLLM text provider for outline and prompt plan
        +-- Notebook image provider for slide image generation
        +-- Notebook edit provider for single-slide image editing
        +-- SlideDeckRepository for state, assets, jobs, exports
        |
        v
Slide Deck artifact + editor workspace + PPTX download
```

AIPPT code is a donor implementation, not a runtime subsystem. Compatible models, prompts, validation logic, export helpers, and UI interaction patterns may be ported into NotebookLM-Lite modules. AIPPT's local `config.py`, standalone `api/routes/*` route shape, standalone `web` app state, and IndexedDB persistence are not retained as product boundaries.

### Model Runtime

Slide decks use existing NotebookLM-Lite model profiles:

- `api.models.text_model`: outline generation, prompt plan generation, edit-instruction rewriting, structured JSON repair.
- `api.models.image_model`: slide image generation.
- `api.models.edit_model`: single-slide image editing.

Text planning must use NotebookLM-Lite `LiteLLMProvider.generate_structured(...)` with Pydantic models equivalent to AIPPT's `DeckOutline` and `SlidePromptPlanSet`. This removes AIPPT's independent OpenAI/Gemini text router from the integrated path.

Image generation and image editing must still be configured through NotebookLM-Lite model profiles. If LiteLLM supports the selected image/edit endpoint directly, use that. If the endpoint needs raw OpenAI-compatible multimodal chat behavior, implement it behind a NotebookLM-Lite `ImageGenerationProvider` / `ImageEditProvider` using the same `ModelProfile` fields. Do not add provider-specific PPT config screens.

Local testing may copy values from `/Users/lzj/proj/notebook/new_pro/AIPPT/config.yaml` into NotebookLM-Lite local `config.yaml`. No key should appear in committed files, logs, screenshots, or generated docs.

### Workflow

Use a job-backed workflow. LangGraph is appropriate because this is a multi-step, retryable, user-gated flow; however, user confirmation gates must pause the workflow rather than fake automation.

```text
create_deck
  -> collect_selected_source_context
  -> generate_outline_job
  -> wait_for_outline_confirmation
  -> generate_prompt_plan_job
  -> wait_for_prompt_plan_confirmation
  -> generate_slide_images_job
  -> deck_ready
  -> [single_slide_regenerate | single_slide_edit]*
  -> export_pptx_job
```

The workflow should be resumable. If the browser refreshes after outline generation or halfway through image generation, the deck state must be recoverable from backend records.

### Data Model

Add or extend persistent records for:

- `SlideDeckProject`: ID, title, source IDs, config snapshot, status, current stage, created/updated timestamps.
- `SlideDeckOutline`: structured deck outline, user edits, confirmation timestamp.
- `SlidePromptPlan`: structured per-slide plan, display content, image prompts, user edits, confirmation timestamp.
- `SlideRecord`: slide ID, page number, title, prompt, display content, image asset ID/path, status, error, created/updated timestamps.
- `SlideAsset`: image path, MIME type, size, checksum, width/height when available, source stage, model metadata.
- `SlideEditHistory`: slide ID, instruction, previous asset ID, next asset ID, model metadata, timestamp, error state.
- `SlideDeckExport`: deck ID, format, file path, status, created timestamp, download filename.
- `JobRecord`: job ID, deck ID, stage, status, progress, error, retry count, started/finished timestamps.

These records can be stored through the existing SeekDB repository abstraction, with binary files under ignored output/data directories and only references/checksums persisted.

### API Shape

Use NotebookLM-Lite routes, not AIPPT routes:

- `POST /api/slide-decks`: create deck draft from selected source IDs and generation config.
- `GET /api/slide-decks`: list decks.
- `GET /api/slide-decks/{deck_id}`: get full deck state.
- `POST /api/slide-decks/{deck_id}/outline/jobs`: generate or regenerate outline.
- `PATCH /api/slide-decks/{deck_id}/outline`: save user edits and/or confirm outline.
- `POST /api/slide-decks/{deck_id}/prompt-plan/jobs`: generate prompt plan from confirmed outline.
- `PATCH /api/slide-decks/{deck_id}/prompt-plan`: save user edits and/or confirm prompt plan.
- `POST /api/slide-decks/{deck_id}/generate/jobs`: generate slide images from confirmed prompt plan.
- `POST /api/slide-decks/{deck_id}/slides/{slide_id}/regenerate`: regenerate one slide from its prompt.
- `POST /api/slide-decks/{deck_id}/slides/{slide_id}/edit`: edit one slide image with a user instruction.
- `POST /api/slide-decks/{deck_id}/export/jobs`: export PPTX.
- `GET /api/slide-decks/{deck_id}/download?format=pptx`: download exported file.
- Reuse existing `/api/jobs/{job_id}` if present; otherwise add a generic job status endpoint.

### Frontend UX

The main NotebookLM-Lite workbench remains the default screen:

- Left: sources.
- Middle: chat.
- Right: Studio tools and generated artifacts.

Slide Deck entry points:

- Studio PPT card starts a new deck from selected sources and opens the dedicated editor workspace.
- Existing generated Slide Deck artifact card opens the persisted deck workspace.

The dedicated workspace should preserve NotebookLM-Lite warmth and spacing, but can reuse AIPPT's structure:

```text
┌────────────────────────────────────────────────────────────┐
│ NotebookLM-Lite / Slide Deck       Back to notebook  Export │
├───────────────┬────────────────────────────┬───────────────┤
│ slide list    │ large slide preview        │ workflow/edit │
│ outline steps │ image / loading / errors   │ confirm/edit  │
│ page status   │                            │ regenerate    │
└───────────────┴────────────────────────────┴───────────────┘
```

Required interactions:

- Back button returns to main workbench without losing deck state.
- Outline is editable before confirmation.
- Prompt plan is editable before confirmation.
- Slide list shows status per page.
- Center preview shows selected slide image, loading state, and errors.
- Right panel supports regenerate current slide, edit current slide, view edit history, export PPTX, and download when ready.
- The UI must not expose AIPPT's independent API key form; configuration stays in NotebookLM-Lite settings.

### Export

Phase 1 export is PPTX export produced from generated slide images. Implementation may reuse AIPPT `api/routes/export.py` logic, but it should be moved behind a NotebookLM-Lite export service/job and file store.

Product UI can say "PPTX export". README or technical docs should avoid claiming native editable slide objects. If needed, say "PPTX export with generated slide visuals" or similar.

### Testing Strategy

Testing must include:

- Unit tests for Pydantic slide schemas, model-runtime calls, repository persistence, export helper behavior, and config redaction.
- Backend integration tests for create deck, outline job, outline confirmation, prompt plan job, prompt confirmation, image generation job with mocked provider, single-slide regenerate/edit, export job, and job failure recovery.
- Frontend unit tests for deck workspace state, confirmation UI, return navigation, slide list, edit history, export button, and error handling.
- Browser E2E tests using Playwright or equivalent for mouse-click flows: select sources, click PPT card, generate outline, confirm outline, generate prompt plan, confirm prompt plan, generate slides, edit/regenerate one slide, export PPTX, download.
- Real model smoke test using local NotebookLM-Lite `config.yaml` plus PPT model parameters from `/Users/lzj/proj/notebook/new_pro/AIPPT/config.yaml`. This must generate at least a small deck, show it in the browser, run one single-slide edit or regenerate, export PPTX, and confirm the downloaded file exists and opens/has expected slide count.

## Risks / Trade-offs

- Real image generation is slow and expensive. Mitigate by using a small 2-3 slide deck for smoke tests and keeping larger deck generation as optional manual validation.
- Image/edit provider APIs vary. Mitigate by wrapping them behind NotebookLM-Lite provider interfaces and keeping fixtures for raw OpenAI-compatible multimodal responses.
- Source context can exceed model limits. Mitigate by summarizing/retrieving selected chunks before deck planning and storing the context snapshot used for generation.
- Browser refresh during generation can lose local state if state remains frontend-only. Mitigate by persisting deck state and polling job status from backend.
- AIPPT's frontend IndexedDB project model is convenient but conflicts with NotebookLM-Lite persistence. Mitigate by porting interaction structure, not storage ownership.
- PPTX output is image-based. Mitigate with accurate wording and defer native editable PPTX to a later change.

2. Port Pydantic schemas and prompt templates from `new_pro/AIPPT` into NotebookLM-Lite backend modules.
3. Implement model providers and slide deck services behind NotebookLM-Lite config.
4. Implement persistent deck/project/job/export records.
5. Add routes and mocked integration tests.
6. Add the dedicated frontend workspace and connect it to real backend routes.
7. Replace PPT placeholder behavior with real entry/open behavior.
8. Run mock tests, builds, browser smoke tests, and real-model smoke tests.
9. Update README/config docs and remove or keep any old placeholder code only if marked as legacy.

## Open Questions

- Whether PDF export should be included in Phase 1 along with PPTX. AIPPT supports it, but the user explicitly prioritized PPTX export.
- Whether generated deck source context should be a frozen snapshot only, or whether the deck should also remember live source IDs for future regeneration against updated sources. The safer Phase 1 behavior is both: freeze the context snapshot used for generation and keep source IDs as lineage.
- Whether the slide workspace should use a route such as `/decks/:deckId` or internal app state. A route is preferable for reload/recovery and shareable local navigation.
